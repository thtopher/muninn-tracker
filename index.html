The vision: ODIN answers analytical questions in natural language against Starset's BigQuery data — so instead of someone writing SQL manually, the system handles it. The semantic layer defines what ODIN can answer, how much it costs, and where it breaks. Everything in this timeline builds toward that.

Overall Progress

P0 P1 P2 P3 P4

P0: Architecture & Guardrails Mostly Complete

Decided the architecture (BigQuery + Gemini + GCP), estimated costs, defined the team, and built the sprint plan. Everything that had to be true before we could start building.

P1: Semantic Foundation Active

Build the "analytical contract" — what ODIN can answer and how. Collect the real questions from SMEs, map them to BigQuery, build semantic views, and validate that the system actually works.

P2: Controlled Interaction Preliminary

P1 proves the semantic layer works technically. P2 answers the harder question: does it work well enough that a real person would trust the answer? SMEs and stakeholders grade response quality and surface the gaps that only real usage reveals.

P3: Platform Transition Preliminary

P2 validated the answers. P3 builds the product around them. ODIN goes from "an internal tool Courtney runs" to "a thing clients can log into and use." Web app, auth, per-client isolation, cost controls. Sean is critical.

P4: Operationalization Preliminary

P3 built the product. P4 puts it in front of real clients and finds out if it delivers value. Pilot deployments, real usage data, legal frameworks, pricing validation, and the operational muscle to support production.

Phase Gates

Each gate is a go/no-go decision. If the conditions aren't met, we don't proceed — we fix what's broken first.
